## 问题：被标记为局部最优的卡组有大量相同（或只相差一个物品），和预期不符

### 现象简述

在报告中「问题 1」和「局部最优去重与相似度」小节可以看到：

- **局部最优条目总数**（按 ComboSig）往往远大于**唯一物品组合数**（按物品集合去重）。
- 同一组物品、不同槽位排列会对应多个 ComboSig，每个都可能被标记为局部最优。
- 许多「不同」的局部最优卡组，要么**物品组合完全相同**（仅排列不同），要么**仅差 1 个物品**（邻域内替换得到的另一局部最优）。

若预期是「局部最优 ≈ 少量互不重复的强组合」，则会感到「大量相同或只差一物」与预期不符。

### 原因分析

1. **ComboSig 与排列绑定**
   - 状态里一条「卡组」由 **ComboSig** 唯一标识，而 ComboSig 与**物品 + 槽位形状 + 排列**相关；同一组物品的不同排列会得到不同 ComboSig。
   - 爬山时每个「当前卡组」是某一具体排列；当该排列被判定为局部最优时，只标记**这一条 ComboSig** 为局部最优，不会自动合并其它排列。
   - 因此，**同一物品组合**若被多个锚定/不同起点以不同排列爬到局部最优，就会在报告里出现多条「物品相同、排列不同」的局部最优。

2. **邻域是「同尺寸单卡替换」**
   - 爬山的邻域是「同形状下、换掉一张同尺寸的牌」。若卡组 A 是局部最优，则其「换一张牌」得到的邻居 B 若也被某次爬山独立地爬到并判定为局部最优，就会产生**仅差 1 个物品**的另一条局部最优。
   - 不同锚定玩家、不同赛季、不同起点都可能爬到这些相邻组合，导致大量「差一物」的局部最优对。

3. **没有「按物品组合去重」的合并逻辑**
   - 当前设计是「每条 ComboSig 独立标记 IsLocalOptimum」，不会在写入或报告时按「物品集合」合并为一条代表。因此统计上会自然出现「大量相同或只差一物」。

### 与预期的关系

- **若预期**：局部最优 = 「互不重复的强组合」的少量代表 → 当前实现**不会**自动做这层去重，所以会看到大量重复与「差一物」。
- **若预期**：局部最优 = 「所有被爬山判定为无法单步改进的 ComboSig」的忠实记录 → 当前行为符合；重复与差一物是**多起点、多排列、邻域关系**的必然结果。

### 建议（算法/产品层面）

1. **报告与展示层做去重**
   - 在**展示**「局部最优卡组」时，按**物品集合**去重，只保留每组一个代表（如 ELO 最高或对局数最多），避免给用户造成「满屏同一套牌」的印象。脚本报告中的「局部最优去重与相似度」已提供唯一组合数与重复分布，便于做这类展示或二次分析。

2. **存储层是否合并**
   - 若希望状态里「局部最优」更精简，可在写入时增加逻辑：同一物品组合只保留一条代表（如保留 ELO 最高的一条 ComboSig 的 IsLocalOptimum），其余同组合的条目不再标为局部最优。这会改变状态语义（从「每条 ComboSig 独立」变为「每物品组合一条代表」），需与「断点续跑、历史池」等用法一致。

3. **邻域与「差一物」**
   - 「仅差 1 个物品」的局部最优对是邻域结构的自然结果，不一定是 bug。若希望减少这类成对出现，可考虑：在判定局部最优后，对「与已标记局部最优仅差一物的当前组合」做特殊处理（例如不再标记为新的局部最优，或合并到同一代表下）；这会引入额外规则与实现成本，需权衡。

4. **预期对齐**
   - 将产品/设计预期明确为二者之一：（a）「局部最优 = 所有无法单步改进的 ComboSig」→ 保持现状，在报告/展示层去重；（b）「局部最优 = 按物品组合唯一的代表」→ 在写入或展示时增加按组合去重/代表选取逻辑，并在文档中说明。
