# Bazaar Arena 优质卡组的查找方案

整体方案采用基于ELO的随机对战模式。每秒大约进行200场对战（基准）。生成大量虚拟玩家在对战中进行学习和调整卡组，从而找到优质卡组。

## 目标

- 验证卡组的绝对强度，给出绝对强度的卡组推荐榜单。
- 为每个物品提供“该物品所在的”卡组强度推荐。

## 约束
- 每个卡组（组合）对应唯一的最强排列。
- 如果一个卡组通过同尺寸物品替换可以取得更高的强度，说明它并非当前优化方法的完全体，不应该作为卡组推荐的终点。

## 优化方案

- **虚拟赛季**：优化器的执行每一轮称为一个“虚拟赛季”。
- **形状**：每名虚拟玩家都固定了一种卡组形状，即卡组中小、中、大物品的数量。对于6个槽位而言，卡组形状有600、410、220、030、301、111、002共7种；对于10个槽位而言，卡组形状有14种。虚拟玩家在调整优化自己的build时，只会进行同大小卡牌的替换，即卡组形状始终保持不变。
- **启动和注入**：在优化器启动时，引入两类虚拟玩家：第一类玩家（锚定类）是某个物品的“真爱粉”，它们在升级build的过程中永远不会放弃自己所选定的物品；每个物品的每个卡组形状都具有自己的一个锚定类玩家。第二类玩家（强度类）追求强度，在升级build的过程中可能会替换任何物品。之后，优化器每隔若干个虚拟赛季，会引入新的强度类玩家（注入）。玩家引入的时候卡组都是完全随机的，锚定类玩家至少会拥有锚定的物品。
- **ELO天梯**：优化器使用ELO天梯系统，虚拟玩家初始均为1500分数。ELO天梯系统包含1400-1600、1600-1800、1800-2000、2000-2200和2200+五个段位，每个段位包含1个历史玩家池，其中的历史玩家卡组具有数量上限（例如50）。每个虚拟赛季结束时，各个虚拟玩家的卡组都有机会进入当前段位的历史玩家池。
- **卡组相似度**：为了避免相似的卡组过多影响评估公正性，历史玩家池会根据卡组相似度（按照交并比计算）进行调整，在超过数量上限时优先踢出相似的卡组。由于虚拟玩家自身会维护较强的卡组，所以应该优先踢出更早加入历史玩家池的卡组。
- **卡组中的物品次序**：当拿到一个卡组时，虚拟玩家总是会通过内战调整自己卡组中的物品次序到最优解，然后再开始参加比赛。物品自身的邻居偏好标签会帮助找到更加合理的次序。采用激进策略，根据内战最优和邻居偏好直接确定最优，不再需要外部验证。在内战时由于卡组强度接近容易平局，所以不但要关注比赛结果，在平局时还需判定生命较高的一方（虽然都为0或0以下）更强。
- **匹配赛**：在每一个虚拟赛季，一个虚拟玩家会随机匹配到自己段位或者相近段位的虚拟玩家或历史玩家进行对战模拟，从而让所有卡组都能动态变化ELO积分。匹配次数会受到单赛季对局次数上限和单赛季失败次数上限两个方面限制，让弱的卡组及时停止匹配区尝试优化。
- **卡组优化**：在匹配赛之后，每个虚拟玩家还会尝试对自己的卡组进行优化。具体而言，它们会尝试将卡组中的一些物品替换成同尺寸的其他物品。新卡组在确定物品次序之后，会首先和原卡组进行几次内战，如果通过内战检验，则尝试用新卡组进行匹配。如果新卡组取得了更好的ELO，虚拟玩家在下一赛季会切换到新卡组，并停止本赛季的卡组优化尝试。第二类玩家如果切换到其他第二类玩家相同的卡组，则踢出一个玩家（合并优化）。
- **物品替换规则**：在卡组优化选择替换物品时，会参考以下几个方面：单卡强度（在ELO的过程中不断加、减更新，相当于“胜负值”）、和当前卡组已有物品（不包括即将被替换的物品）的上下游配合能力。第一类玩家会倾向于选择自己锚定的物品的配合能力，而第二类玩家会综合考虑已有物品的配合能力。除了显式标注的上下游之外，还有两项启发式规则：所有Ammo物品都具有上游Reload（可以被装填），所有Burn物品都具有上游Burn（灼烧累计）。
- **放弃**：如果一个虚拟玩家长期无法得到卡组优化，可能是其卡组已经达到了局部最优，这种情况下我们将其标记为放弃，不再尝试参加后续赛季的对局。第二类玩家放弃时会直接被踢出（结束优化），仅保留其卡组在历史玩家中。第一类玩家会暂时标记该卡组为其锚定物品的“最优卡组”，然后再重新从包含该物品的随机卡组开始，以寻求其他优化路线。

## 输入输出
- 输入可以用命令行参数或者 JSON 文件。
- 保存当前优化状态到JSON文件，支持从文件开始继续优化。除此之外还包括优化中的一些元数据，以在优化效果较差时分析问题。
- 控制台上输出两个部分：一是第二类玩家的TOP10卡组（表示当前找到的最强卡组），二是随机抽取10个物品，每个物品选取ELO分最高的第一类玩家，输出这些玩家的卡组（表示当前找到的这些物品所在的最强卡组）。

## 小规模测试与效果检查
- 使用 `--max-seasons 10` 可只跑 10 个虚拟赛季后自动退出并保存状态（适合快速验证）。
- 运行结束后，用脚本根据保存的元数据检查探测效果：
  `python scripts/inspect_quality_deck_state.py <状态文件路径>`
- 脚本会输出：当前赛季、总对局数、池内卡组数、ELO 范围、强度/锚定玩家数，以及按 ELO 排序的 Top 10 卡组（含物品列表与对局数），便于评估探测器是否在有限赛季内产生合理分化与高 ELO 卡组。
